* dotfiles
  inspired by https://github.com/mathiasbynens/dotfiles

  A simple setup to automate and document the setup of my development machine
  as much as reasonable.

  Full automation doesn't make a lot of sense as it is a lot of work,
  needs to be adjusted every now and then,
  and won't be used much more often than every few years.
  The invested time just doesn't justify.

  But at least this framework gives some tools to recreate a development setup
  on a new machine with not too much effort.

  Currently this repo is optimized for macs.
  But it is also possible to adjust it for usage with different Linux distributions.
  Even a parallel setup and synchronization between a Linux and a MacOS system proved maintainable in the past.

* Concepts
  - The dotfiles are stored in git. They can be checked out to any folder on the target workstation.
  - Via symlinks the dotfiles are "activated".
  - The repo is copied into place via ~bootstrap.sh~ (rsync) instead of managing per-file symlinks.
  - Complex configuration setups from 3rd parties are managed via [[https://git-scm.com/book/en/v2/Git-Tools-Submodules][git submodules]].

** Home Manager (MVP)
   This repo now also contains an experimental standalone [[https://github.com/nix-community/home-manager][Home Manager]] setup under ~nix/~.
   It is intentionally narrow.
   For now it only manages ~~/.p10k.zsh~ and does not replace the broader shell setup in ~.zshrc~ or ~.zprofile~.

*** Supported targets
   - ~gunnar-dotfiles-darwin~ → current macOS architecture (~aarch64-darwin~)
   - ~gunnar-dotfiles-arch~ → Arch Linux (~x86_64-linux~)

*** Prerequisites
   - Install Nix first.
   - The repo-managed user-level Nix config now lives at ~nix/config/nix.conf~.
     It still needs to end up at ~~/.config/nix/nix.conf~ on your machine, for example via symlink or your normal dotfiles sync process.
   - Enable flakes in the live ~~/.config/nix/nix.conf~ or in ~~/etc/nix/nix.conf~:
     #+BEGIN_SRC conf
       experimental-features = nix-command flakes
     #+END_SRC
   - If you are testing local changes, make sure newly added flake files are tracked by git before running flake commands.

*** Build before switch
   Validate the configuration without activation first:
   #+BEGIN_SRC sh
     nix run github:nix-community/home-manager/release-25.11 -- build --flake ./nix#gunnar-dotfiles-darwin
   #+END_SRC

   On Arch Linux, use the Arch target instead:
   #+BEGIN_SRC sh
     nix run github:nix-community/home-manager/release-25.11 -- build --flake ./nix#gunnar-dotfiles-arch
   #+END_SRC

   Activate it only after the build succeeds:
   #+BEGIN_SRC sh
     nix run github:nix-community/home-manager/release-25.11 -- switch --flake ./nix#gunnar-dotfiles-darwin
   #+END_SRC

   On Arch Linux, use the Arch target instead:
   #+BEGIN_SRC sh
     nix run github:nix-community/home-manager/release-25.11 -- switch --flake ./nix#gunnar-dotfiles-arch
   #+END_SRC

*** Collision handling
   Home Manager refuses to overwrite an unmanaged target.
   If ~~/.p10k.zsh~ already exists, either move it aside first or ask Home Manager to keep a backup during the first switch:
   #+BEGIN_SRC sh
     nix run github:nix-community/home-manager/release-25.11 -- switch -b backup --flake ./nix#gunnar-dotfiles-darwin
   #+END_SRC

*** Rollback
   To stop using this MVP setup:
   - stop activating the relevant Home Manager target
   - remove the Home Manager-managed ~~/.p10k.zsh~ symlink or generation output
   - restore the previous ~~/.p10k.zsh~ from the backup you created during first activation

   The backup-based first activation flow above is the intended rollback path for this MVP.

* Essential Tools
** bootstrap.sh
  Copies the tracked configuration into ~$HOME~ using ~rsync~.

  Dry run first (recommended):
  #+BEGIN_SRC
    ./bootstrap.sh -n
  #+END_SRC

  Apply the changes once the output looks right:
  #+BEGIN_SRC
    ./bootstrap.sh
  #+END_SRC

** Local LLM (Ollama)
  Run powerful local models for coding and chat.

  Quick start:
  #+BEGIN_SRC
    bin/ollama-install.sh --yes --start
    bin/ollama-coder.sh --repl              # interactive coder chat (qwen2.5-coder:7b)
    bin/ollama-coder.sh "Write a Bash script..."
  #+END_SRC

  Tips:
  - Use ~-m qwen2.5-coder:14b~ for higher quality if you have RAM/VRAM.
  - General models (non-coding): ~llama3.1:8b~, ~qwen2.5:7b~.
  - Server host: set ~OLLAMA_HOST~ or pass ~--host~ to scripts.
